# P3 — 微信公众号替代采集 via RSSHub SCF

**Date**: 2026-07-18
**Status**: Approved design, awaiting implementation
**Author**: Claude (brainstorming with 飞哥)
**Replaces**: CLAUDE.md "WeChat public accounts have no public API and are out of scope for MVP"

## Context

Project has 24 working sources aggregating global nuclear intelligence. The
信源汇总 spreadsheet (rows 60-68) lists 9 high-value Chinese WeChat public
accounts that are not currently scraped — they have no public API and are
out of scope per CLAUDE.md MVP.

飞哥 elected to unblock this by adopting **RSSHub as a bridge** deployed to
a Tencent Cloud Serverless Cloud Function (SCF) inside mainland China. RSSHub
exposes `/wechat/:mpID` routes that return RSS XML, scraping WeChat via its
internal channels (avoiding the need for a first-party API).

Two hard problems had to be solved at the design level:

1. **WeChat blocks overseas IPs** — Actions runners in `ubuntu-latest` cannot
   reach WeChat content directly. RSSHub must run on a Chinese mainland IP
   surface. SCF provides that without persistent infrastructure.
2. **WeChat route IDs are mpIDs, not names** — RSSHub `/wechat/:mpID` requires
   the numeric mpID (`mp.weixin.qq.com/s?__biz=...`). We need a one-time
   discovery step that resolves names → mpIDs via RSSHub's `/newwechat`
   route (Sogou WeChat backed), then caches the result.

## Decisions

| # | Decision | Rationale |
|---|---|---|
| D1 | RSSHub hosted on Tencent SCF | Lowest cost (pay-per-call, ~¥10/mo) + domestic IP + no persistent infra to maintain. Alternatives (rsshub.app public, self-hosted ECS) have higher cost or lower reliability for this use case. |
| D2 | One-time mpID discovery + git-tracked cache | mpIDs are stable identifiers (公众号 rename doesn't change mpID). Running discovery on every fetch would trigger Sogou WeChat anti-bot and burn rate limits. Cache to `data/wechat_mpids.json` and commit. |
| D3 | mpID cache committed to git | Maintains a review trail for maintainers; PR can catch wrong/duplicated mpIDs; aligns with existing `data/` "regenerated by workflow, but source-of-truth entries are versioned" pattern. |
| D4 | `RSSHUB_BASE` injected via GitHub Actions secret | Avoids committing the URL to git history; allows maintainer to rotate or replace the instance without code changes. |
| D5 | Reuse existing RSS fetcher path | Adding 9 wechat entries is just 9 lines in `NUCLEAR_RSS_FEEDS`. Zero new fetch code. The whole "RSSHub" layer is invisible to `update_news.py` — it only sees `xml_url` strings. |
| D6 | Silent-zero fallback when RSSHUB_BASE missing or RSSHub down | Matches TerraPower / Oklo / DOE-NE pattern. Wrapper records `ok=True, warning=..., items=0`. Pipeline never crashes; source-status.json surfaces the problem for ops. |
| D7 | Hard-fail per-entry on RSSHub HTTP error | Per-entry RSSHub failure (e.g. one mpID returns 404) raises so wrapper records `ok=False, error=...`. Distinguishes per-source failures from systemic ones (no RSSHUB_BASE). |
| D8 | Discovery script is a standalone CLI, not part of pipeline | Discover runs locally with `RSSHUB_BASE` env set; produces JSON; committed. Workflow never runs discovery. This prevents the Actions runner from hammering Sogou WeChat and getting our RSSHub instance rate-limited. |

## Architecture

```
  Local (one-time)              CI (every 30 min)                      SCF (mainland China)
  ──────────────                ──────────────────                     ─────────────────────
  discover script                                                             ┌──────────┐
    python scripts/                                                             │  RSSHub  │
    discover_wechat_mpids.py                                                    │ instance │
      ↓ (env RSSHUB_BASE)                                                       │ (SCF)    │
    data/wechat_mpids.json ──commit + push──>  checkout                          │          │
    (9 rows: name, mpID,                ↓                                      │  routes: │
     fetched_at)            NUCLEAR_RSS_FEEDS entries                            │ /wechat/ │
                            construct xml_url =                                 │ :mpID    │
                            f"{RSSHUB_BASE}/wechat/{mpID}"  ───HTTP GET──>       │          │
                                                      ↓                          │ /newwechat│
                            existing fetch_single_rss_feed() ──>  RSS XML        │ /:name   │

                                                      ↓
                                              data/items.json
                                                      ↓
                                              data/source-status.json
                                              (wechat_* entries
                                               silent-zero or ok=False)
```

## Components

### C1. `scripts/discover_wechat_mpids.py`

Standalone CLI script, run by maintainer locally (never in CI):

```bash
RSSHUB_BASE=https://service-xxxx.gz.apigw.tencentcs.com python scripts/discover_wechat_mpids.py
```

Behavior:
- Reads 9 name seeds from a hardcoded constant `WECHAT_ACCOUNTS` (matching
  the spreadsheet rows 60-68)
- For each name, POSTs to `{RSSHUB_BASE}/newwechat/{name}` (or GET, depending
  on RSSHub version) — see Open Question OQ1
- Parses first mpID match from response
- Writes `data/wechat_mpids.json`:
  ```json
  [
    {"name": "中国核电网 (CNNP)", "mpID": "Mp-xxxxxxxxxxxxxxxxx", "fetched_at": "2026-07-18T..."},
    ...
  ]
  ```
- Exit 0 if all 9 resolved; exit 1 with failed names listed if any failed

### C2. `scripts/deploy_rsshub_scf/`

One-shot deploy bundle, based on RSSHub official `scf-template`:
- `serverless.yml` — SCF config (region, memory, timeout, env vars)
- `package.json` — RSSHub version pin
- `deploy_rsshub_scf.sh` — wrapper that runs `sls deploy` with proper region/auth

Maintainer runs locally once:
```bash
cd scripts/deploy_rsshub_scf
npm install
./deploy_rsshub_scf.sh
# outputs: https://service-xxxx.gz.apigw.tencentcs.com/release/rsshub
# copy URL into GitHub Actions secret RSSHUB_BASE
```

### C3. `data/wechat_mpids.json` (git-tracked)

9 entries; same structure as in C1. Committed by maintainer after running
discovery. update_news.py reads this file at fetch time.

### C4. NUCLEAR_RSS_FEEDS — 9 new entries in `scripts/update_news.py`

```python
{
    "site_id": "wechat_cnnp",
    "site_name": "中国核电网",
    "xml_url": "{RSSHUB_BASE}/wechat/{mpID}",   # resolved at fetch time
    "html_url": "https://mp.weixin.qq.com/...",
    "_mpID_key": "cnnp",                        # key into wechat_mpids.json
    "via_jina": False,
}
```

`xml_url` is a **template string** resolved by a new helper
`_resolve_wechat_xml_url(site_id, mpids_map, rsshub_base)` that runs at fetch
start. If `rsshub_base` is empty / `mpids_map` is missing the entry, the
fetcher short-circuits to silent zero (matches D6).

### C5. `scripts/update_news.py` changes

- New helper `_resolve_wechat_xml_url(site_id, mpids_map, rsshub_base) -> str | None`
- New constant `WECHAT_ACCOUNTS: tuple[dict, ...]` — 9 entries with site_id /
  display name / mpID lookup key (mirrors spreadsheet rows 60-68)
- New helper `_load_wechat_mpids() -> dict[str, str] | None` — reads
  `data/wechat_mpids.json`, returns `{mpID_key: mpID}` map or None if file
  missing/malformed
- New env read `RSSHUB_BASE = os.environ.get("RSSHUB_BASE", "")`
- New helper `_fetch_wechat_rss(session, feed_def, mpids_map, rsshub_base, now)`
  wraps existing RSS path; if rsshub_base empty → return []; else resolve
  template URL and call existing `_fetch_rss_xml`

## Data Flow (operational sequence)

**Initial setup (one-time)**:
1. Maintainer clones repo, installs deps
2. Deploys RSSHub to SCF (C2) → gets URL
3. Sets `RSSHUB_BASE` in GitHub Actions secrets
4. Runs `discover_wechat_mpids.py` locally → produces `wechat_mpids.json`
5. Commits + pushes `wechat_mpids.json`
6. Next scheduled workflow run picks up all 9 entries

**Per-run (every 30 min)**:
1. Workflow starts, env `RSSHUB_BASE` injected
2. `update_news.py` reads `wechat_mpids.json` → 9 entries resolved
3. For each, fetches RSS XML → parses → nuclear_kw_score → dedup
4. Failures surface in `source-status.json` with appropriate ok/warning/error

**mpID rotation (rare, manual)**:
1. Maintainer observes one entry going silent-zero repeatedly (likely mpID stale)
2. Re-runs `discover_wechat_mpids.py --only <site_id>` to refresh that one row
3. Commits updated JSON

## Error Handling

| Failure point | Behavior | Visibility |
|---|---|---|
| `RSSHUB_BASE` secret missing | All wechat_* fetch → silent zero | source-status.json: `ok=True, warning="RSSHUB_BASE not configured"` |
| `wechat_mpids.json` missing/malformed | All wechat_* fetch → silent zero | source-status.json: `ok=True, warning="wechat_mpids.json missing"` |
| Single mpID stale (RSSHub 404) | Single entry fetch → raise → wrapper → `ok=False` | source-status.json: `ok=False, error=...` |
| RSSHub instance 5xx | Single entry fetch → raise → wrapper → `ok=False` (per-entry); other entries still try | source-status.json: per-entry `ok=False`; if all 9 fail with same error, ops sees systemic issue |
| WeChat rate-limited | RSSHub returns 429; entry → `ok=False` with rate-limit hint | source-status.json: per-entry error message |
| Sogou WeChat anti-bot on discover | discover script fails to resolve some names; exit 1 with failed names listed | Maintainer sees local error; no commit; ops unaffected |

**Rationale for split (silent-zero vs hard-fail)**: When the *bridge*
(RSSHUB_BASE / mpID cache) is missing, the gap is operational and applies to
all 9 sources uniformly → silent-zero with one warning is clearer than 9
identical errors. When the bridge is up but a *specific mpID* is stale, that
one entry deserves explicit per-entry error to surface "which公众号 needs
refresh".

## Scope and Out of Scope

**In scope**:
- 9 named公众号 from spreadsheet rows 60-68 (中国核电网 / 中核集团 / 中国核建 / 中国能源研究 / 核闻 / 核电那些事 / 国家核安全局 / 上海核电 / 中国核动力研究设计院)
- Tencent SCF as the RSSHub host (single deploy bundle)
- Discovery script + cache file + 9 NUCLEAR_RSS_FEEDS entries
- Silent-zero + per-entry hard-fail error handling
- GitHub Actions secret injection
- Tests (4 modules, ~12 tests)

**Out of scope** (explicit YAGNI):
- Multi-cloud / multi-instance RSSHub (single SCF only)
- Auto-rotation of mpIDs (maintainer refreshes manually when needed)
- Per-entry discovery (no fallback chain; single source of truth)
- 微信公众号 beyond the 9 named (extending list = new design cycle)
- RSSHub content transformation (titles/cleaning) beyond what RSSHub emits
- Subscription-tier mpIDs that RSSHub `/newwechat` cannot resolve (will be
  reported as discovery failures; not solved in this iteration)

## Open Questions

**OQ1**: RSSHub `/newwechat/:name` — HTTP method (GET vs POST) and response
shape (HTML scraped from Sogou vs JSON API) varies across RSSHub versions.
Need to probe against the specific RSSHub version pinned in `package.json`
before writing the parser. If parsing shape is brittle, alternative is
running Sogou WeChat HTML scraping locally and feeding mpIDs by hand.

**OQ2**: SCF cold start — RSSHub on cold start can take 5-15s. Within 30-min
workflow tolerance, fine. But if SCF times out at default 3s, we need to
raise the timeout in `serverless.yml`. Will check during initial deploy.

**OQ3**: WeChat 公众号 article retention — articles older than ~weeks drop
out of mp.weixin.qq.com listings. RSSHub may cache differently. Acceptable
since `RSS_MAX_AGE_DAYS=14` already filters. No action needed.

**OQ4**: Some公众号 may be subscription-only (付费 / 企业认证). RSSHub
`/newwechat` may not resolve them. discover script will report these as
failures; ops can either drop the entry or find another bridge (not in scope).
